1.0       Summary

 

1.1            Introduction

In an effort to standardize the data loading and hosting process for the FDIC DMS project, the Data Loading Coordinators (DLC) team has created this document to outline several key standards and procedures. Bank teams should adhere to the guidelines expressed in this document to bring consistency to our data captures, avoid defects in loading, and heighten the quality of our deliverables.

 

1.2            Intended Audience

The intended audience for this document includes the following:

           DMS Oversight Managers

           DMS Bank Capture Site Leads

           DMS Bank Capture Team Members

 

 

2.0            Extraction Standards

2.1            Extract File Naming Convention

         Prefix extracts names with bank abbreviation, followed by vendor name and application name

o   <<BANK ABBREVIATION>>_<<VENDOR NAME>>_<<APPLICATION NAME>>_<<CATEGORY>>_<<SUB CATEGORY>>_ <<DESCRIPTION>>.txt

         For Bank Abbreviation please reach out to your respective DLC in the first week of capture

         Here is an example for Integrity Florida Fidelity BancPac Loan Master

o   i.e., IF_Fidelity_BancPac_Loan_Master.txt

         If multiple files exist for a single extract, suffix it with a sequential number

o   i.e., IF_Fidelity_BancPac_Loan_Master_1.txt, IF_Fidelity_BancPac_Loan_Master_2.txt, etc.

         Extract file names should not contain / \ ? * or characters

         Extract name should closely resemble iCONECT Database Names

         In the Data Extracts Collection on the individual extract tabs, when you enter the Extract Name in cell B3 DO NOT include the bank abbreviation or DB number

o   i.e., Fidelity BancPac Loan Master

 

2.2            Database Naming Standard

         Begin with the 2 letter bank abbreviation, followed by vendor name and application name before the full extract description

o   Bank Abbrev + Vendor Name + Application Name + Category + (Subcategory + Description) + DB Number

o   i.e., CL Fidelity Online Loan Collection Transaction History 107

         The limit for Database Names is 64 characters

         Spell out Vendor Names and Application Names

         Capitalize the first letter of each word in the database name (like a book title)

         Database names should not contain underscores, commas, or parenthesis

o   Use a forward slash / instead of commas

o   Use curly braces {} instead of parenthesis

         Please refer to 4.2 Database Naming Standard Examples and 4.3 Database Naming Standard FAQ for more details

 

2.3            Field Naming Convention

         Field names have a maximum length of twenty characters

o   Field names that are pulled from application database column names should be expanded to be more descriptive because they usually contain unnecessary abbreviations

o   Do not abbreviate field names unless they are longer than twenty characters or you are following the Business QA field name standards

o   Field names should be descriptive and easy for the end user to understand

         Boolean fields should NOT include Y/N in the field name. Instead, any such field should include the word Flag in the column header

o   Example: Loan Payment (Y/N) should be Loan Payment Flag

 

2.4            File Saving Convention

         All extracts (.txt files) should be saved in ANSI format, not Unicode

         Use WordPad or Text Pad in order to check for the format. WordPad will not open a file that is not ANSI formatted.

 

2.5            iConect Data Types

Following are the only data types allowed in the extracts collection:

         To identify data types of respective fields please follow the steps below:

  Profile the data extracted (all or a reasonable size sample)

  After profiling, any field with limited text size less than 200 characters, update it to 200 in length

         FULL TEXT Limit of 2 GB

         LIMITED TEXT Length between 200 - 4000 characters

o   All the account number fields (Loan Number, DEP Number, GL Account Number, Customer Number, SSN Tax ID, etc.) should be limited text

o   Zip codes, phone numbers, and other non-calculation fields should be captured and maintained as limited text

o   Data in iCONECT should exactly match the raw data. Example A field with value 00817 and specified as Number in the DEC will get loaded as 817 in iCONECT. The right thing to do here will be load this field as limited text so that it gets loaded as-is (which is 00817).  Any field that could potentially have a value beginning with a leading 0 must be loaded as limited text.

o   Fields containing numbers that would be loaded with a scale of 0 (i.e., integers with no decimal values) must be loaded as limited text.  Some example fields include ID, File Size, Batch Number, Sequence Number, Routing Number, etc.

o   For fields 200 characters and above, double the length

o   Exception: File Path fields do not follow this rule. They need to be specified as Limited Text fields with length between 255-4,000 characters in the Data Extracts Collections document. File Path field should include the absolute path for the attachment.

o   File Link should also be a Limited Text field with 255 characters. The File Link field should be the SAN path (link) for the report.

         NUMBER/NUMERIC (p, q) Number fields should only be used for rate and amount fields (i.e. fields which are considered calculation fields)

o   Amount/currency fields by default should be loaded with a scale of 2.  However, make sure you profile the data and ensure no records contain more than 2 digits after the decimal place.

o   Percentage/rate fields by default should be loaded with a scale of 5.  However, make sure you profile the data and decrease the scale when appropriate if no records contain values with more than 2 or 3 decimal places.  Make the scale longer if some values contain more than 5 decimal places.

o   p is the length and q is the scale. Ensure that length <= 38 and scale <= length-1. Length includes the digits before and after decimal and the decimal point (i.e., 100.05 has length=6, scale=2).

         DATE MM/DD/YYYY format

o   Month and Day can be one or two digits (04/03 and 4/3 are equivalent)

o   Year must be in four digits

o   If the date was originally stored as MM/DD/YY, the data should NOT be reformatted to MM/DD/YYYY. Instead, it should be maintained in MM/DD/YY format and treated as a Limited Text field. If this is the case, it must be listed as an issue on the Known Issues tab of the DEC.

2.6            Field Names

         Field names cannot be longer than 20 characters

         Maximum number of fields allowed per extract is 500

         Duplicate field names are not allowed

         Do not use underscores

         When appropriate, follow the naming conventions from the Business QA

         Field name prefixes (LN, CUST, etc.) are only required on fields from the Business QA and do not need to be added to all fields this should help save space and allow you to spell out more abbreviations

         Capitalize the first letter of each word in the field names (like a book title)

 

2.7            Lockheed Martin Testing

         Every extract must pass LM Testing prior to leaving bank site

         Create intelligent sample extracts (should not exceed 10,000 records)

         Run the Extracts Testing Program on all Data Extracts Collections prior to sample load testing: https://er43.deloitteonline.com/eRoom/FDICASTEP/FDIC-Internal/0_ef05

         Run the DMS Extracts Analyzer on each extract to uncover commonly encountered data issues: https://er43.deloitteonline.com/eRoom/FDICASTEP/FDIC-Internal/0_797c

         Run the Link Checker on each extract containing links: https://er43.deloitteonline.com/eRoom/FDICASTEP/FDIC-Internal/0_eb4b

         Upload sample extracts and associated Data Extracts Collections in a zip file under the LM Testing folder in the e-room when ready for testing

         Maintain and update the Database Key document to track load testing status

 

2.8            File Encryption

         Consultants with Windows XP on their laptops have DFE, Deloitte File Encryption, installed.  If a DMS hard drive is connected to these laptops, the laptop idle time will trigger DFE to begin encryption of the DMS hard drive. 

         The encrypted files are green and only that user and that laptop can decrypt the files. 
Right-click on a green file/properties/advanced in order to retrieve the user information that encrypted the file.

         Any DMS practitioner with XP should first disable DFE prior to connecting a DMS hard drive.  From task manager/processes, stop the Deloitte file encryption process.  Note that this will be activated again once the laptop is re-booted.

 

2.9            Extract Data Dos and Donts

         First record in every extract file should be the field header (pipe delimited)

         Pipe character (|) should be used as delimiter in ALL extracts. Since pipe is used as a delimiter, remove any pipe characters present in the data.

         Data should not contain double quotes

         Every extract should contain a Record Source column as the last field

o   << APPLICATION NAME >> / <<DATABASE NAME>> / <<KEY ENTITY NAME>>

o   i.e., Application Name/Database/Entity

         Remove any $ or % characters in numeric fields

         Zip Code and SSN fields should be set as Limited Text, not Number because they could start with a 0 (which is truncated by iCONECT for Number fields)

         Do not remove fields that have null values for all records. These fields should be maintained to preserve data integrity as the DMS team is not able to determine what data is critical for litigation.

o   If the number of fields exceeds 500, create a supplemental extract to maintain the rest of the fields.

3.3            Rockville Folder

Data captured at the bank sites should not be manipulated during the course of loading at Rockville. All the extracts go through a rigorous cycle of sample testing and so it is not common to encounter situations where an extract file needs to be fixed in order to load successfully into iCONECT. If such a need arises, please follow the steps outlined below:

         Create a folder called Rockville on an external drive

         Create a copy of the original extract and move it to this folder

         Address the data issues off of this copy, and do not overwrite the file that was created at the bank site

Please note the following:

         Create this folder on an external drive and maintain only one copy of this folder

         Maintain a log of all the changes (issues, fix etc.) in this folder for all the files that were fixed in order to load them successfully into iCONECT

         Ensure that this folder is also copied to the SAN by Lockheed Martin before exiting Rockville

         All final versions of all deliverables should also be stored on the external eRoom prior to leaving Rockville after the load is complete

4.3 Database Naming Standard FAQ

1)      What should I do if I run out of space?

It is highly unlikely that you will need more than 64 characters but it is possible. Try the following in the order listed before escalating it to your site manager and the DLC team:

         Abbreviate commonly known banking acronyms.

         Abbreviate vendor names

         Remove the vendor name

2)      Should I list the vendor name twice if it is same as the application name?

List only once. Also, vendor name is not applicable for custom developed applications.

3)      Do I always need to have a description?

In some cases, a combination of category and subcategory should be sufficient (Loan Master, Customer Master, etc.) and so a description is not always applicable.

5.0            Interim Servicing Data Captures

There are two scenarios for Interim Servicing Data Captures. Please see the outline of each scenario below for further information.

         Scenario One: Interim data capture is performed by the DMS team

All standard capture documents must be created except for the original DAM which will be updated. For scope, use the interim scope addendum template (available in e-Room).

o   Naming Convention: the interim capture documents will have the original file names suffixed with the word interim. For example:

  DMS America West Bank Initial Cert Interim.xlsx

  DMS America West Bank Data Extraction Certification Document Interim.doc

         Data loading for Interim Servicing depends on whether it is follow-up for an existing database from the original capture or the extract is new and does not align with an existing database

o   Existing Data will have the same name as the original database with a suffix of the letter I. This ensures the two databases will be next to each other in iCONECT. For example:

  Original DB Name: AW Fiserv Director Reports 15

  Interim DB Name: AW Fiserv Director Reports 15I

o   New Databases that do not align with any existing database will be created in iCONECT using the original two letter abbreviation and will follow, in consecutive order, the last database that was created from the original capture for that bank.

  If the last DB had the number of 17, the interim servicing database will have DB number 18 followed by the suffix I

         Scenario Two: Data is received from the onsite PCAM team

o   Receive the completed Interim Data Request Fulfillment form

o   Load the extracts as per scenario 1

o   Update the original DAM

6.1            Deliverables Overview

Each bank is responsible for a collection of client deliverables to be handed over to Lockheed Martin and the FDIC upon completion of the extraction process. The following deliverables are due to the client:

         Scope Document

         Initial Certification

         Data Access Manual

         Data Extract Collection

         QA Document

         Extraction Certification Document

 

These documents should be polished and finalized by the time the extraction completes at the bank-site.

 

6.2            Scope Document Guidelines

The scope document represents the contractual agreement with the FDIC. It is the most important of the client deliverables that the DMS team produces. The scope document is created and amended over the course of the extraction lifecycle, starting with the Remote Assessment. The initial version of the scope document produced during this phase receives sign-off from the client. Once onsite, the bank team has the ability to amend the scope document as the extraction evolves. The scope document is considered a living document throughout this process, and it should always be updated to accurately reflect the data which is extracted from the bank.

Factors in Creating and Amending a Scope Document

         Traceability

o   All items within the DMS capture scope should be enumerated to allow for traceability across client deliverables

o   Include extraction method (i.e. if vendor assistance is required)

         Accuracy

o   The description and volumetric information for each scope item should be as accurate as possible. A bulleted list of what data was housed in each application should be included in the Description/Functional Overview of the scope document

o   Volumetric information should closely match the volumetric information of the DEC and QA documents

o   Date ranges should be reconciled to the actual extracts

o   Make all updates to the document that are necessary after the stakeholder review
